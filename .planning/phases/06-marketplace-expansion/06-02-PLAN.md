    Per D-02: Unified ESM Trading API at etapi.ebaykorea.com. Single adapter class serves both Gmarket and Auction. Per D-05: Use iconv-lite for EUC-KR encoding if needed.

    **client.ts**: Create `createEsmClient(apiKey: string)` function.
    - Base URL: `https://etapi.ebaykorea.com` (ESM Trading API)
    - Use ky.create() with prefixUrl
    - beforeRequest hook: set `Authorization` header to `Bearer {apiKey}` (ESM uses API key as bearer token)
    - Content-Type: `application/json;charset=UTF-8` (ESM Trading API uses JSON, not XML -- despite legacy Gmarket/Auction being XML-based, the unified ESM API is modern JSON)
    - Timeout 30s, retry on 408/429/500/502/503/504
    - If ESM API actually returns XML for some endpoints, import fast-xml-parser as fallback. Per D-05 use iconv-lite if EUC-KR encoding detected.

    **types.ts**: Define ESM Trading API response types.
    - `EsmApiResponse<T>`: { resultCode: string, resultMessage: string, data: T } (standard ESM wrapper)
    - `EsmOrder`: orderNo (string), siteType ('G' for Gmarket, 'A' for Auction), orderItemSeq (string), itemName (string), orderQty (number), buyerName (string), buyerPhone (string), receiverName (string), receiverPhone (string), receiverZipcode (string), receiverAddress (string), receiverAddressDetail (string), orderDate (string), orderStatus (string), sellPrice (number), payAmount (number)
    - `EsmClaim`: claimNo (string), orderNo (string), siteType (string), claimType (string -- CANCEL/RETURN/EXCHANGE), claimStatus (string), claimReason (string), claimDate (string)
    - `EsmProduct`: for getProducts
    - `EsmInvoicePayload`: orderNo, orderItemSeq, deliveryCompanyCode, invoiceNo

    **status-map.ts**: Create status mapping.
    - `ESM_STATUS_MAP`: Map ESM status codes to OrderStatus
      - 'ORDER_RECEIVED' -> 'new'
      - 'PAYMENT_COMPLETE' -> 'new'
      - 'PRODUCT_PREPARE' -> 'preparing'
      - 'DELIVERING' -> 'shipped'
      - 'DELIVERY_COMPLETE' -> 'delivered'
    - `mapEsmStatus(code: string): OrderStatus` -- fallback 'new' with console.warn
    - `ESM_CLAIM_STATUS_MAP` and `mapEsmClaimStatus()` / `mapEsmClaimType()` functions
    - Same fallback pattern as Coupang

    **adapter.ts**: Create `EsmAdapter` class implementing MarketplaceAdapter.
    - Constructor takes `{ api_key: string, site_type: 'G' | 'A' }` -- site_type determines if this instance serves Gmarket ('G') or Auction ('A')
    - Config is dynamically set based on site_type:
      - 'G': id='gmarket', name='지마켓', authType='api_key', rateLimitPerSecond=30, requiredCredentials=['api_key']
      - 'A': id='auction', name='옥션', authType='api_key', rateLimitPerSecond=30, requiredCredentials=['api_key']
    - `testConnection()`: GET a lightweight endpoint to verify API key
    - `authenticate()`: return { success: true } (API key auth, no separate flow)
    - `getOrders(since)`: GET /api/v1/orders with date range + siteType filter, normalize to NormalizedOrder[]. Set marketplaceId based on site_type ('gmarket' or 'auction').
    - `getClaimsOrders(since)`: GET /api/v1/claims with siteType filter, normalize to NormalizedClaim[]
    - `uploadInvoice(orderId, invoice)`: POST /api/v1/orders/{orderId}/delivery with carrier code (mapCarrierCode(this.config.id, invoice.carrierId)) and tracking number
    - `getProducts()`: GET /api/v1/products with siteType filter, normalize to NormalizedProduct[]
    - `registerProduct(product)`: POST /api/v1/products
    - `updateProduct(id, product)`: PUT /api/v1/products/{id}
    - Error handling: same pattern as Coupang -- MarketplaceAuthError on 401/403, MarketplaceApiError on other failures
    - Private normalizeOrder(), normalizeClaim() methods following Coupang pattern
