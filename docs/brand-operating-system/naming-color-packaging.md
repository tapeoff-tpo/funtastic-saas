# FUN-TASTIC Naming, Color and Packaging Rules

Checked: 2026-07-18

## Naming Architecture

Use a stable descriptive hierarchy:

`FUN-TASTIC / Collection / Module Family / Job or Fit / Size or Orientation / Generation`

Customer-facing names should remain short. Catalog metadata carries technical detail.

### Naming rules

- Lead with the job or familiar object, not internal connector terminology.
- Use one canonical name per module family.
- Universal modules receive Zone-neutral names.
- Object-fit variants name the compatible object explicitly and conservatively.
- Size terms must have a defined catalog meaning.
- Generation appears only when compatibility or performance meaningfully changes.
- Color and edition never replace the functional product name.
- Korean and English names should preserve the same hierarchy and meaning.
- Avoid names confusingly similar to established modular, toy, or lifestyle systems.
- Legal and trademark review is required before public adoption.

### Example grammar, not approved names

- Core / Tray / Small / Generation 1
- Work / Headphone Hanger / Standard / Generation 1
- Personal / Label Plate / Name / Generation 1

These examples describe grammar only; no product naming is finalized.

## Compatibility Language

Every package and product page should state:

- Interface generation
- Compatible Base classes
- Zone and environment class
- Orientation or load class where necessary
- Included parts
- Required service part, if any
- Explicit exclusions

Use a consistent compatibility mark, but do not invent visual certification before legal and user testing.

## Color Strategy

### Layer 1: Anchor neutrals

- Stable Base and evergreen module colors
- Designed to coexist with interiors and later accents
- Limited in number to protect batch and inventory control

### Layer 2: Core accents

- A small permanent family of pastel and vivid colors
- Selected for composition, recognition, photography, and availability
- Must be reproducible across approved materials and suppliers

### Layer 3: Functional accents

- Optional cues for orientation, category, grip, status, or service parts
- Never the only compatibility signal

### Layer 4: Seasonal palettes

- Time-bounded expression on approved evergreen products
- Up to two controlled palette windows per year as an initial governance limit
- No essential function restricted to a seasonal color

### Layer 5: Collaboration palettes

- Rights-cleared and audience-relevant
- Must still work with the core neutral and compatibility system

No Pantone, RAL, NCS, or filament values are fixed in this strategy phase. Physical standards require CMF prototyping, supplier capability, photography, aging, and cleaning validation.

## Color Governance

- Each approved color receives an ID, master reference, material scope, photography reference, and lifecycle state.
- Similar shades require a portfolio reason, not personal preference.
- Color names remain stable across releases.
- Retired colors are archived and not silently renamed.
- Batch variation and screen-to-physical mismatch are disclosed internally and controlled.
- Color variants are tracked separately from unique functional modules in reporting.

## Packaging Job

Packaging has five responsibilities:

1. Show the current customer job.
2. Explain attachment in the shortest understandable sequence.
3. Prove compatibility and limits.
4. Identify included modules and service parts.
5. Reveal the next relevant expansion without making the present product feel incomplete.

## Packaging Hierarchy

### Front

- Product and module-family name
- One real use image or clear installed state
- Compatibility generation and Base class
- Included quantity

### Back or secondary panel

- Attach, confirm, remove
- Environment and load boundaries
- Included parts
- Cleaning and service guidance
- One or two adjacent compatible modules

### Digital layer

- QR may link to current instructions, compatibility, service, and configuration examples
- Critical safety or compatibility information must not exist only behind a QR code

## Packaging Rules

- No opaque blind packaging for core functional modules.
- Avoid excessive internal terminology.
- Do not present a render as proven installed performance.
- Do not imply wet, heat, load, food-contact, child-safety, or third-party compatibility without evidence.
- Package structure should protect the product and support fulfillment before decorative complexity.
- Sustainability claims require evidence; material reduction alone is not a complete claim.
- Edition packaging must preserve core compatibility and service information.

## Naming and Packaging Test

Before approval, a participant who has not seen the system should be able to identify:

- What the product does
- What it connects to
- Whether it suits their Zone
- What is included
- How to begin attachment
- Which next module is relevant

Success thresholds must be set during actual comprehension testing.
