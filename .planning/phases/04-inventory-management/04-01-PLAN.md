<task type="auto">
  <name>Task 2: Stock adjustment actions and order status hooks</name>
  <files>src/lib/inventory/actions.ts, src/lib/orders/actions.ts</files>
  <read_first>src/lib/db/schema.ts, src/lib/orders/actions.ts, src/lib/inventory/types.ts, src/lib/inventory/queries.ts</read_first>

1. Create `src/lib/inventory/actions.ts` with these functions:

   `adjustStock(userId: string, sku: string, delta: number, reason: AdjustmentReason, opts?: { note?: string, orderId?: string })`:
   - Uses db.transaction with SELECT FOR UPDATE on inventory row (per D-02)
   - If no inventory record exists for (userId, sku), return error (do NOT auto-create -- admin must create inventory items first via setStock)
   - Updates totalStock += delta, recalculates availableStock = totalStock - reservedStock
   - Inserts inventoryHistory record with previousTotal, newTotal, delta, reason, note, orderId
   - Returns { success: true, newTotal } or { success: false, error }

   `setStock(userId: string, sku: string, productName: string, totalStock: number)`:
   - UPSERT: inserts new inventory record or updates existing
   - Sets totalStock and recalculates availableStock = totalStock - reservedStock
   - Creates inventoryHistory entry with reason 'physical_count' and delta = newTotal - previousTotal
   - Returns { success: true }

   `deductForOrder(tx: DrizzleTransaction, userId: string, orderId: string)`:
   - Called INSIDE an existing transaction (receives tx parameter)
   - Queries order_items for the given orderId to get SKU + quantity pairs
   - For each item with a non-null SKU: adjustStock-like logic (within tx) to decrement totalStock by quantity, reason 'order_ship'
   - Items without SKU are silently skipped (not all order items may be mapped to inventory yet)
   - Does NOT fail the whole operation if inventory record doesn't exist for a SKU -- logs warning and continues

   `restoreForOrder(tx: DrizzleTransaction, userId: string, orderId: string)`:
   - Same pattern as deductForOrder but increments totalStock, reason 'order_cancel' or 'return'

2. Modify `src/lib/orders/actions.ts` -- `updateOrderStatus`:
   - After the successful status update within the existing transaction:
   - If newStatus === 'shipped': call `deductForOrder(tx, order.userId, orderId)` (per D-02, D-03)
   - If newStatus === 'cancelled' AND previous status was 'shipped' or 'delivering': call `restoreForOrder(tx, order.userId, orderId)` (per D-03)
   - Import userId from the order row (already selected in the transaction)
   - Add userId to the select fields in the existing transaction query

   Note: The order row already uses SELECT FOR UPDATE. The inventory deduct/restore happens in the SAME transaction, ensuring atomicity. Do NOT create a separate transaction.

3. Also add a hook for claims: export `restoreForClaim(userId: string, orderId: string)` from inventory/actions.ts that can be called when a return claim completes. This is a standalone function (not inside a tx) since claims processing is separate from order status changes.
