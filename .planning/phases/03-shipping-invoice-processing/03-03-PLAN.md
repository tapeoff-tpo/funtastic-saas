    - Test: 2 orders with same buyer but different addresses -> 0 merge groups (each has only 1 order)
    - Test: 3 orders same buyer/address, but 1 has fulfillmentCode 'frozen' and 2 have 'normal' -> 1 merge group with 2 normal orders (frozen is separate, but only 1 frozen order so no group)
    - Test: 5 orders same buyer/address, maxPackQuantity=3 -> 2 groups (3+2)
    - Test: normalizeAddress strips whitespace, normalizes 시/도/구 variations
    - Test: Single order with same buyer/address -> no merge group (need 2+ to merge)
    - Test: Orders from different days -> separate groups even if same buyer/address

  <action>
    Create `src/lib/shipping/combined-shipping.ts` per D-05, D-06, D-07:

    1. `normalizeAddress(address: { zipCode: string, address1: string, address2?: string }): string`
       - Trim whitespace, remove extra spaces, lowercase
       - Return `${zipCode}|${address1}|${address2 || ''}`

    2. `getFulfillmentCode(orderItems: Array<{ fulfillmentCode?: string }>): string`
       - If all items have same fulfillmentCode, return it
       - If mixed, return 'mixed' (will be treated as separate group)
       - Default to 'normal' if no fulfillmentCode set

    3. `findMergeCandidates(orders: OrderWithItems[], maxPackQuantity: number = 10): ShipmentGroup[]`
       Algorithm per research Pattern 2:
       a. Group by `buyerName + normalizeAddress(shippingAddress) + format(orderedAt, 'yyyy-MM-dd')`
       b. Filter to groups with 2+ orders
       c. Within each group, sub-group by `getFulfillmentCode(order.items)`
       d. Within each sub-group, if count > maxPackQuantity, chunk into groups of maxPackQuantity
       e. Only include sub-groups with 2+ orders as merge candidates
       f. Return array of ShipmentGroup with groupKey, orders, fulfillmentCode, suggestedAction='merge'

    Input type: `OrderWithItems` = order row joined with items (use type from schema inference). The function is pure -- takes data, returns groups. No DB access.
  </action>

  </verify>
