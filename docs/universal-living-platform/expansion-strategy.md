# FUN-TASTIC Universal Living Platform - Expansion Strategy

Checked: 2026-07-18

## Expansion Thesis

The platform should expand through proven customer transitions, not by filling a theoretical 1000-SKU spreadsheet.

The preferred sequence is:

`one paid job -> adjacent module -> second Zone -> new Base class -> new material -> curated long tail`

## Portfolio Axes

The 1000-SKU capacity comes from seven controlled axes:

| Axis | Long-term capacity | Purpose |
|---|---:|---|
| Base and mounting classes | 20 | Dry, mounted, portable, wet, expansion and service variants |
| Universal module families | 80 | Trays, cups, hooks, clips, dividers, labels, stands and sizes |
| Zone-core modules | 390 | Thirty validated module SKUs across each of thirteen Zones |
| Object-fit and size variants | 200 | Devices, tools, collections and routine-specific fit |
| Validated color/material variants | 150 | Demand-led CMF and environment variants |
| Personalization and collaborations | 100 | Names, teams, creators, seasonal and limited editions |
| Adapters, replacements and service | 60 | Compatibility, repair and migration |
| **Total logical capacity** | **1000** | Not an instruction to stock all items |

## Roadmap

The milestone data is in `1000-sku-roadmap.csv`.

### Stage 1: 10 SKUs

- One dry-zone Starter Kit
- One Base
- Universal modules that solve Study and Work jobs
- Replacement and blanking parts
- No wet-zone or high-load promise

### Stage 2: 30 SKUs

- Add Bedroom as the first cross-Zone proof
- Test Beauty or Living Room according to paid demand
- Measure whether customers reuse modules rather than merely buying another organizer

### Stage 3: 100 SKUs

- Cover six dry Zones
- Add object-fit modules only where actual fit requests repeat
- Establish catalog metadata, compatibility labels, versioning, and retirement policy

### Stage 4: 250 SKUs

- Pilot Coffee-dry, Entryway, Pet, and Travel
- Introduce mounted and portable Base classes only after separate testing
- Test whether a second Base accelerates module purchases

### Stage 5: 500 SKUs

- Introduce PETG or another validated material
- Pilot Bathroom, Kitchen-wet, and Laundry-humid
- Keep the shared interface while changing environmental Base and module material

### Stage 6: 1000 SKUs

- Expand the long tail through on-demand production
- Retain only modules with clear use, margin, compatibility, and support ownership
- Use collaborations and personalization without letting them replace core utility

## Required Metrics

The following must be measured by cohort and Zone:

- Starter Kit conversion
- Base-to-module attach rate
- Time to first follow-up module order
- Second-Zone adoption
- Average modules per active Base
- Module reuse or relocation between Zones
- Compatibility-related return and support rate
- Print yield, labor, packing time, and contribution margin
- Small follow-up order shipping economics
- Active versus dormant module rate
- Interface-version migration cost

Thresholds are not invented in this report. Management must set them from actual unit economics and pilot behavior.

## SKU Governance

- A color, size, bundle, replacement, and object fit must be distinguishable in the catalog data.
- Every module needs an owner, evidence source, compatibility version, and retirement state.
- On-demand variants should not all become pre-stocked inventory.
- Low-demand modules can remain digital manufacturing recipes if service time stays acceptable.
- Redundant modules should be merged rather than preserved to inflate catalog size.
- Retired modules must not silently disappear when customers depend on them.

## Expansion Kill Conditions

Pause Zone expansion when:

- Follow-up orders are mainly replacement purchases caused by failure.
- Customers buy separate kits but do not reuse the interface.
- Compatibility support consumes the margin of small orders.
- Shipping makes single-module replenishment irrational.
- A Zone requires claims beyond the validated material or Base class.
- The catalog is growing faster than active usage.

## 1000-SKU Feasibility

**Technically and catalogically feasible, commercially unverified.** FDM and a shared interface can support a long tail, but 1000 stocked finished goods would create excessive color, fit, packaging, discovery, and support complexity. The viable model is a small stocked core plus on-demand modules, controlled variants, and explicit catalog governance.
