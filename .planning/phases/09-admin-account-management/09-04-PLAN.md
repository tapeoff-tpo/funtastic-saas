1. **Modify `src/app/(auth)/layout.tsx`** to additionally look up the caller's profile and redirect to `/login?reason=deactivated` if `deactivatedAt` is set:

   ```tsx
   import { redirect } from 'next/navigation'
   import { eq } from 'drizzle-orm'
   import { createClient } from '@/lib/supabase/server'
   import { db } from '@/lib/db'
   import { userProfiles } from '@/lib/db/schema'
   import { AppShell } from '@/components/layout/app-shell'

   export default async function AuthLayout({ children }: { children: React.ReactNode }) {
     const supabase = await createClient()
     const { data: { user } } = await supabase.auth.getUser()
     if (!user) redirect('/login')

     const [profile] = await db
       .select({ deactivatedAt: userProfiles.deactivatedAt })
       .from(userProfiles)
       .where(eq(userProfiles.id, user.id))
       .limit(1)

     if (!profile) {
       // No profile row — likely a stale auth.users entry that was never backfilled.
       // Sign out defensively.
       await supabase.auth.signOut()
       redirect('/login?reason=no_profile')
     }
     if (profile.deactivatedAt) {
       await supabase.auth.signOut()
       redirect('/login?reason=deactivated')
     }

     return <AppShell>{children}</AppShell>
   }
   ```

   **Performance note:** this adds ONE indexed lookup per RSC render. The `user_profiles_role_active_idx` from Plan 01 covers the common case. At this scale (≤20 profiles, owner-only access pattern) impact is negligible.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cookie JWT → Server Component render | RSC uses `getUser()` which validates the JWT against Supabase. The added profile lookup verifies the in-app deactivation state. |
| Server-side redirect → Browser | `redirect()` issues a 307; client follows. No data leak — the gate page never renders for unauthorized users. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-22 | Elevation of Privilege | Admin user discovers /admin/accounts via direct URL | mitigate | Layout-level gate at `(auth)/admin/accounts/layout.tsx` redirects non-super_admin to `/`. Page-level gate (Plan 03) is the second line of defense. |
| T-09-23 | Spoofing | Deactivated user with still-valid JWT continues using app | mitigate | Layout-level deactivation check fires on every authenticated request. Pair with `ban_duration` set at auth server for redundancy. |
